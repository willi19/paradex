Distributed robot data pipeline

Paradex is a system for recording and processing multi-camera robot experiments.

It coordinates Capture PCs, camera services, hardware-triggered cameras, robot and hand streams, calibration, and post-processing so each session can be checked for pose and grasp tasks.

Pipeline stages

Each stage depends on the one before it.

Start with configuration and service checks that do not move hardware. Move to calibration, recording, and robot-dependent checks only after the lower layers are known to work.

  1. 01 Select rig profile

    Choose the active system/current profile; verify camera serials, PC names, ports, and shared storage.

  2. 02 Verify services

    Check Python imports, SSH launch, command/data transport, camera services, and hardware-trigger timing.

  3. 03 Calibrate geometry

    Solve intrinsics, extrinsics, and hand-eye transforms when camera-to-robot geometry is required.

  4. 04 Record session

    Record synchronized camera, robot, hand, and teleoperation streams into shared storage.

  5. 05 Post-process data

    Match timestamps, undistort frames, encode videos, reconstruct scenes, upload, and prepare datasets.

  6. 06 Inspect results

    Check pose estimates, grasp transforms, calibration overlays, visualizers, and robot readiness.

Machine roles

Which computer runs what

Paradex separates orchestration from hardware ownership. The Main PC coordinates work; Capture PCs own camera hardware; robot control runs only where the robot drivers are installed and the robot is reachable.

Main PC

Runs orchestration scripts, validation entry points, remote camera commands, monitoring, and processing.

  • src/validate/data_sender/main.py
  • src/capture/camera/*_remote.py
  • src/process/**

Capture PCs

Run camera services that own the FLIR/PySpin camera lifecycle and return frame/status data.

  • src/camera/server_daemon.py
  • src/camera/monitor_daemon.py
  • camera clients launched over SSH

Robot/control host

Runs arm and hand drivers, teleoperation, kinematics, and motion checks when hardware is connected.

  • src/capture/robot/**
  • src/validate/robot/**
  • paradex.io.robot_controller

Shared storage

Stores calibration files, raw sessions, processed datasets, uploads, and reusable caches.

  • ~/shared_data
  • camera params and processed videos
  • NAS-backed processing results

Core terms

Terms used throughout the system

These terms match the code and the physical setup. Use them consistently when reading scripts, logs, and generated datasets.

Rig profile
The selected robot/camera setup under system/current, including PCs, cameras, network settings, and storage paths.
Capture PC
A machine connected to one or more FLIR cameras. It runs camera services and usually receives commands from the Main PC.
Camera service
The long-running process on a Capture PC, such as server_daemon.py, that owns the camera SDK and responds to remote commands.
Hardware trigger
A shared timing signal, typically from the UTGE900, that makes cameras expose frames on the same pulse.
Session
One recorded experiment folder containing raw camera data and any robot, hand, teleoperation, timestamp, and calibration snapshots.
Intrinsic calibration
Per-camera lens and sensor parameters needed for undistortion and projection.
Extrinsic calibration
Camera-to-camera or camera-to-scene geometry used to relate views in one coordinate frame.
Hand-eye calibration
The camera-to-robot transform used when camera observations must align with robot motion or robot-frame overlays.

Setup checks

First checks for a new rig profile

Run these in order before recording a long session. The first checks do not move hardware; later checks depend on camera services and calibration.

1. Install package and select rig profile

git clone https://github.com/willi19/paradex.git
cd paradex
pip install -e .

# confirm the active rig profile and PC map
ls -l system/current
cat system/current/pc.json

2. Start Capture PC camera services

# on every capture PC
python src/camera/server_daemon.py

# optional status page on each capture PC
python src/camera/monitor_daemon.py

3. Verify transport and camera control

# on the main PC
python src/validate/data_sender/main.py
python src/validate/command_sender/stream_remote.py
python src/validate/camera_system/remote_camera_controller.py

4. Continue to calibration and recording

python src/calibration/intrinsic/capture.py
python src/calibration/extrinsic/capture.py
# this upstream hand-eye script currently supports xArm
python src/calibration/handeye/capture.py --arm xarm

# c starts recording, s stops recording, q exits
python src/capture/camera/video_remote.py --save_path dataset/001 --sync_mode
